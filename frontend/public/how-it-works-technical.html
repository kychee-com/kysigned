How it works: the technical details

A kysigned signature is an email its sender's provider signed with DKIM. Nothing about the proof depends on kysigned staying online: the trust set is the signer's email provider, the public DKIM-key archives, and the timestamp authorities.

1. The document and the cover

The creator uploads a PDF. For each signer, kysigned prepends a one-page cover (the ESIGN/UETA and eIDAS consent to sign electronically, plus the document's SHA-256) and computes a per-signer canonical PDF P = cover ++ document. Each signer is emailed their own P as an attachment.

2. The signature is a DKIM-signed forward

A signer signs by forwarding that email back with I sign this document as the first line, keeping the attachment. Their provider (Gmail, Outlook, iCloud, and so on) adds a DKIM signature over the message headers and body hash on the way out. Only that provider can produce a valid DKIM signature for that mailbox, so the forward is a signature only the signer could have made. Signers need no account and no app.

3. Classical verification, then evidence capture

kysigned verifies the DKIM signature with the mainstream mailauth library, confirms the forwarded attachment is byte-identical to the P that was sent, checks the signing phrase on the first line, and records the signer. The raw signed .eml is time-stamped with RFC 3161 and OpenTimestamps.

4. The evidence bundle

When every signer is in, kysigned assembles a single self-contained PDF: the original document, each signer's cover, each signer's original DKIM-signed .eml, the RFC 3161 + OpenTimestamps proofs, and the DKIM public keys with their public-archive provenance. It is a clean, unsigned PDF (no certificate dictionary), so it opens with no warning in any viewer, and it carries a SHA-256 fingerprint over the embedded evidence printed on the signature page. Every party is emailed the bundle; the working copy on kysigned's servers is then deleted.

5. Verification is client-side and offline

Anyone can verify the bundle with no account and no server call. The verifier recomputes each signer's P = cover ++ document from the embedded parts, checks that it matches the signed .eml attachment, validates each DKIM signature against the embedded and publicly-archived keys, validates the time proofs, and recomputes the printed fingerprint. A clean verdict means: these signers, this document, at this time, provable forever. A reference command-line verifier and a second independent cross-check ship in the repository.

Want to see a verdict before signing anything? The repository ships sample files under docs/test-assets/: a genuine signed record for /verify, tampered bundles that must fail, and original + sign-request pairs for /hashcheck, each with a documented expected verdict.
